LESS

Starting with Edna, and Buzz, we are using the LESS preprocessor to write our CSS. Preprocessors add a lot of new things to basic CSS such as nesting selectors, the use of variables and mixins, which are similar to Javascript functions.

Why LESS? We started using LESS and not SASS because of the HLM project. For that we used Bootstrap, which was built with LESS.

Dangers

LESS can make it very easy to take just a few lines of LESS CSS and, when compiled, end up with thousands of lines of CSS. Yipes! In order to avoid this some guidelines should be followed.

Of course one might not be able to help not following a guideline or two. Just make sure that what is compiled is clean and concise.

File Structure

The file structure of Edna is fairly simple. We have one directory, less, that contains all our LESS files. Our main files live in the root of Edna. We have three LESS files there, one for the bulk of Edna, one minified version of that, and one file just for IE. These get compiled into files with the same name, but different suffixes. 

edna
├── less
│   ├── _autocomplete.less
│   ├── _button-group.less
│   ├── _button.less
│   └── _carousel.less
│   └── etc.
├── edna.css

├── edna.less
├── edna.min.less
└── edna.ie.less

Nesting

Nesting is one of the fantastic things included in LESS, but like mentioned above, one can quickly create a ruleset with a ton of selectors, which is not a good thing. One should use better specificity rather than another selector, or !important.

Our first guideline is to keep the number of stringed selectors to no more than 3

Let's look at some nesting examples

Toggle this example
/* THIS IS AN EXAMPLE OF GOOD NESTING */

// here is an example of a nested LESS ruleset
.example {
    foo: bar;
    .sub-example {
        foo: bar;
    }
}

// and here is the output
.example {
  foo: bar;
}
.example .sub-example {
  foo: bar;
